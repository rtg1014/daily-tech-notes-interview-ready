## 과정설명


- 예를들어 사용자가 www.google.com을 입력하면, 브라우저는 HTTP 프로토콜을 사용해 구글 웹 서버와 통신하려고 합니다. 
- HTTP는 OSI 7계층 중 애플리케이션 계층에서 동작하는 프로토콜입니다.

<br>

- 이때 브라우저는 요청한 도메인 이름(www.google.com)에 대한 IP 주소를 알아야 하기 때문에 DNS(Domain Name System) 서버에 질의합니다.
- 이 질의 과정 또한 애플리케이션 계층에서 이루어지며, DNS 서버는 해당 도메인에 대한 IP 주소(예를 들어, 142.250.190.78)를 응답합니다.

<br>

- IP 주소를 얻은 후, 브라우저는 구글 서버와 통신을 시작합니다. HTTP는 TCP/IP를 기반으로 작동하므로, 데이터를 주고받기 전에 TCP 3-Way Handshake 과정이 필요합니다.
- 이 단계는 전송 계층(4계층) 에서 이루어집니다.


<br>


- TCP 연결이 성립된 후, 브라우저는 HTTP Request 메시지를 생성하여 구글 서버에 보냅니다.
- 예를 들어, 브라우저는 “GET / HTTP/1.1”이라는 요청을 TCP 프로토콜을 통해 80번 포트로 전송합니다. 이때 데이터는 패킷(Packet) 형태로 네트워크를 통해 전달됩니다.
- 네트워크를 통해 데이터를 전송하기 위해서는 네트워크 계층(3계층) 에서 IP 주소를 사용하고, 데이터 링크 계층(2계층) 에서 MAC 주소를 사용하여 패킷이 전송됩니다.

<br>



- 구글 서버는 클라이언트의 요청을 수신하고 이를 처리한 후, HTTP Response 메시지를 생성하여 응답합니다.
- 서버는 요청이 성공했음을 알리는 200 OK 상태 코드와 함께 웹 페이지 데이터를 전송합니다.
- 브라우저는 이 응답을 받아 HTML, CSS, 자바스크립트 등의 데이터를 해석하여 화면에 페이지를 렌더링합니다.

<br>

- 모든 데이터 전송이 완료되면 클라이언트와 서버는 4-Way Handshake 과정을 거쳐 TCP 연결을 종료합니다.

## 🚀 **사용자가 웹사이트에 처음 접근했을 때 발생하는 과정 (Step-by-Step 분석)**  

웹사이트에 처음 접속하는 과정은 **단순히 브라우저에서 주소(URL)를 입력하는 것**처럼 보이지만, 실제로는 **여러 단계의 네트워크 통신과 데이터 로딩, 렌더링 과정**이 필요합니다.  

기술 면접에서는 단순한 과정 나열이 아니라 **“왜 이런 과정이 필요한가?”**, **“어떤 방식이 최적의 성능을 보장하는가?”** 등을 깊이 있게 이해하고 있어야 합니다.  

아래에서는 **사용자가 웹사이트에 처음 접속했을 때** 발생하는 과정들을 **일상적인 비유**와 함께 풀어보겠습니다.  

---

## 🛠 **1. 사용자가 웹사이트 URL을 입력하면 무슨 일이 일어날까?**
💡 **비유:** 웹사이트를 방문하는 과정은 마치 **"처음 가는 음식점을 찾는 과정"**과 비슷합니다.  
1️⃣ 먼저, 음식점의 이름(도메인)을 알고 있어야 하고  
2️⃣ 지도(네트워크)를 통해 정확한 위치(IP)를 찾아야 하며  
3️⃣ 도착 후 메뉴(HTML, CSS, JS)를 받고  
4️⃣ 음식을 조리하고(렌더링)  
5️⃣ 손님이 음식을 즐기는 것(사용자 경험)과 유사합니다.  

---

## 🌎 **2. 도메인 네임 시스템(DNS) 조회 - 웹사이트 주소 찾기**
**🔹 과정:**  
- 사용자가 `www.example.com`을 입력하면, 브라우저는 **DNS 서버**에 해당 도메인의 **IP 주소**를 요청합니다.  
- 만약 브라우저가 최근에 이 주소를 방문한 적이 있다면 **캐시된 IP 주소**를 사용합니다.  
- DNS 서버는 `example.com`이 **어떤 IP 주소(예: 192.168.1.10)와 연결되어 있는지** 알려줍니다.  

**💡 비유:**  
📍 **레스토랑 찾기** → 식당 이름(도메인)을 알고 있지만, 정확한 주소(IP)를 모르기 때문에 네이버 지도(DNS)에 검색하여 위치를 찾는 과정과 같습니다.  

**⚠️ 심화 질문:**  
1️⃣ **DNS 조회 과정에서 TTL(Time To Live)이 무엇인가요?**  
   - DNS는 캐싱을 활용해 **불필요한 요청을 줄이기 위해 TTL 값을 설정**합니다.  
   - 예를 들어 TTL이 300초(5분)라면, 같은 요청이 5분 동안은 DNS 서버에 다시 요청하지 않고 **캐시된 데이터를 사용**합니다.  

2️⃣ **DNS Prefetching이란?**  
   - 브라우저가 예상되는 DNS 요청을 미리 실행하여 성능을 최적화하는 기술입니다.  
   - 사용자가 링크를 클릭하기 전에 **미리 해당 도메인의 IP를 조회**하여 빠르게 접속할 수 있도록 합니다.  

---

## 📡 **3. TCP Handshake - 서버와 연결하기**
**🔹 과정:**  
- **DNS 조회 후, 브라우저는 IP 주소를 찾았지만, 아직 서버와 연결된 것이 아님!**  
- 브라우저는 **TCP(Transmission Control Protocol)** 를 사용해 **3-way Handshake** 과정을 거쳐 서버와 연결합니다.  

**📍 과정 (3-Way Handshake)**  
1️⃣ 클라이언트(브라우저) → 서버에 **SYN (Synchronize 요청)** 전송  
2️⃣ 서버 → 클라이언트에게 **SYN-ACK (요청 수락 + 확인 응답)** 전송  
3️⃣ 클라이언트 → 서버에 **ACK (최종 확인 응답)** 전송  
✔️ 연결이 성공적으로 맺어짐!  

**💡 비유:**  
📞 **전화 통화 연결** → 친구와 전화를 할 때도 "여보세요?"(SYN) → "어, 잘 들려!"(SYN-ACK) → "좋아, 말해봐!"(ACK) 하는 과정이 필요하듯, 브라우저와 서버도 통신하기 전 이 과정을 거쳐야 합니다.  

**⚠️ 심화 질문:**  
1️⃣ **왜 TCP Handshake가 필요한가요?**  
   - 클라이언트와 서버 간의 **신뢰할 수 있는 연결을 보장**하기 위해 필요합니다.  
   - 데이터 손실 없이 패킷을 주고받을 수 있도록 보장합니다.  

2️⃣ **HTTP/2 또는 HTTP/3에서는 TCP Handshake 과정이 어떻게 최적화되나요?**  
   - HTTP/2에서는 **Multiplexing** 기술을 이용해 한 번의 연결로 여러 요청을 동시에 처리합니다.  
   - HTTP/3는 **TCP 대신 UDP(QUIC 프로토콜)을 사용**하여 Handshake 단계를 줄이고, 연결 속도를 개선합니다.

## 🔐 **4. TLS Handshake - 보안 연결 설정 (HTTPS)**
**🔹 과정:**  
- 브라우저와 서버 간의 데이터가 **암호화되지 않으면, 해커가 쉽게 정보를 가로챌 수 있음**  
- 이를 방지하기 위해 TLS(Transport Layer Security) 프로토콜을 이용하여 **암호화된 통신**을 설정  

📍 **HTTPS 과정 (TLS Handshake)**  
1️⃣ 브라우저 → 서버에 **"나는 TLS를 사용하고 싶어!"** 라고 요청  
2️⃣ 서버 → **TLS 인증서**(SSL Certificate)를 브라우저에게 전송  
3️⃣ 브라우저 → **공개키를 사용해 비밀 키를 생성** 후, 서버와 공유  
✔️ 이제부터는 모든 데이터가 **암호화된 통신**을 통해 안전하게 전송됨!  

**💡 비유:**  
🔑 **비밀 편지 교환** → TLS는 마치 **비밀스러운 대화**를 위한 암호화된 편지 교환과 같습니다.  
📩 친구와 주고받는 편지를 **특수한 자물쇠(암호화)로 잠가서** 중간에서 누군가 훔쳐봐도 내용을 알 수 없도록 보호하는 방식과 유사합니다.  

**⚠️ 심화 질문:**  
1️⃣ **HTTPS가 HTTP보다 느린 이유는?**  
   - TLS Handshake 과정이 추가되면서 초기 연결 시간이 증가함.  
   - 하지만 **TLS 세션 재사용, HTTP/2, QUIC** 등을 통해 성능 최적화 가능.  

2️⃣ **HSTS (HTTP Strict Transport Security)란?**  
   - 브라우저가 **HTTP 요청을 아예 차단**하고 HTTPS만 사용하도록 강제하는 보안 정책.

## 🏗 **5. HTTP 요청 및 응답 - 웹 페이지 가져오기**
**🔹 과정:**  
1️⃣ **브라우저가 서버에 HTTP 요청** (예: `GET /index.html`)  
2️⃣ 서버는 요청을 처리하고, **HTML, CSS, JS 등의 리소스를 응답으로 전송**  
3️⃣ 브라우저는 HTML을 분석하여 추가적인 리소스 (이미지, 폰트 등)를 요청  

📍 **이때 발생하는 최적화 기법**  
✅ **HTTP Caching** → 동일한 요청을 다시 보내지 않도록 브라우저 캐시 사용  
✅ **Lazy Loading** → 사용자가 필요할 때만 이미지/스크립트 로드  
✅ **CDN (Content Delivery Network)** → 가까운 서버에서 리소스 제공  

**💡 비유:**  
🍽 **요리 재료 배달** → 웹사이트 로딩은 마치 요리 재료를 주문하는 것과 비슷합니다.  
🚚 HTTP 요청 = "필요한 재료 목록을 가게에 요청"  
📦 HTTP 응답 = "요청한 재료를 배달"

## 🎨 **6. 브라우저 렌더링 과정**
✅ HTML 파싱 → DOM 생성  
✅ CSS 파싱 → CSSOM 생성  
✅ JavaScript 실행 → JS가 DOM을 조작  
✅ 최종적으로 **페인트(Paint) & 합성(Composite) 과정** 후 화면에 표시  

**💡 비유:**  
🖌 **그림 그리기** →  
1️⃣ HTML = 밑그림  
2️⃣ CSS = 색칠  
3️⃣ JS = 애니메이션 효과 추가

## 🚀 **정리**
1️⃣ 사용자가 웹사이트 URL 입력 → **DNS 조회 (IP 주소 찾기)**  
2️⃣ **TCP Handshake** → 서버와 안정적인 연결 설정  
3️⃣ **TLS Handshake** → HTTPS 보안 연결 설정  
4️⃣ **HTTP 요청 및 응답** → HTML, CSS, JS 가져오기  
5️⃣ **브라우저 렌더링** → 화면에 페이지 표시  

✅ **기술 면접에서는 단순한 과정 설명이 아니라 "왜 이렇게 되는지?", "최적화 방법은?"을 묻기 때문에, 심화 내용을 이해하는 것이 중요합니다!** 🚀
